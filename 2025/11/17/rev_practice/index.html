漏洞点

在v0没有判断正负，若为负，则可从&unk_4060的地址往前读

可通过修改结构体stdout的内容利用该漏洞。

另外，&unk_4060指针free之后没有置0，指针悬空。

unk_4060可以看作python中的list，有index和content，content是堆的chunk的地址，当free掉index位置的content时，chunk中会存入堆基址等内容，此时show就会显示chunk中的内容。正常来说，free之后应当把unk_4060对应index的content置0，也就是将存的chunk地址置0，才能防止uaf。

通过拿到的堆基址进而泄露libc地址，即可通过environ变量获取栈地址并计算返回地址的偏移，覆写返回地址执行rop，再执行shellcode获取flag。

题目文件：quest、secret_box.exe

题目分析

main函数可以看到程序启动方式./secret_box.exe quest，然后可输入32字符的passcode，然后判断passcode的长度v4，如果长度小于32字节，则退出。

对一些参数重命名，关键函数为vmrun，该函数会对用户输入的passcode和quest的内容进行验证，验证通过才能执行后续RC4解密的过程。

分配了 0x10 的空间同时又 v7 和 v8 做模运算了赋值给 v11 指向的地址。操作完成之后又执行了 *((_QWORD *)v11 + 1) = v2; 和 v2 = v11;，以上过程是链表结构。如果将划分的 0x10 字节内存进行划分，也可以看出，前八个字节存储数据，后八个字节存储指针。

shift+F1 打开 IDA 的 local type 窗口，按 insert 键插入结构体的定义，然后y键v2、v9、v10，将其转化为结构体data，就会变得清晰很多，实际上v2是一个链栈，重命名为stack。

IDA 高级功能使用 之 创建结构体 —— 加快你的逆向速度 - thinkycx.me

也就是说从栈中取出两个值，分别给到v7和v8，那么v11就是从栈中取两个int，计算%，再存入栈中。那么case0即为取模%。v11也是data，value是int型。

继续看case1，涉及v6，v6为pc指针，v3为opcode，显然case1是一个入栈操作push。

恢复结构体之后 vm 还是很一目了然的，下面解释一下各个 opcode 的作用。

  • 0：取模操作（先弹出的值在运算符左侧）
  • 1：push 操作
  • 2：pop 操作
  • 3：乘法操作
  • 4：加法操作
  • 5：输出
  • 6：取输入指针
  • 7：右移位后取最低位（先弹出的值在运算符右侧）
  • 8：异或操作
  • 9：输入指针+1
  • 10：返回
  • 11：减法操作（先弹出的值在运算符左侧）
  • 12：除法操作（先弹出的值在运算符左侧）

可以根据操作实现虚拟机，这里已经很清楚是栈的数据结构了就直接用 std::stack 即可。

其中最明显的应该能看到它频繁的取输入字符并做 (x>>y)&1 的运算，y 从 0~7，不难想到，这是在一个一个取出输入字节的每一位，每一位都对应了一个权值，留在了栈中。第一个字节可以看出来，从低位到高位权值分别为

最后将所有权值相加，得到的结果和 70 做异或运算得到 24，将该值存入栈底。

第一个异或值，在最后被加起来并返回。而外层函数判断用户输入是否正确，依赖于返回值是否为 0，因此要尽可能让每次异或值都相等。

这个VM验证机制实际上是：

  • 将每个输入字符的8个比特位作为选择器
  • 每个比特位选择对应的质数权重
  • 计算加权和：sum = Σ(bit_i × weight_i)
  • 验证加权和是否等于目标值

可以将要输出的值打印到 stderr 中，再用重定向 2>out.txt 就可以快速拿到一些值。可以考虑拿出case3乘法中的权值、case8异或的目标值（^符左侧的值）打印到stderr中

然后写反推的脚本，是单字节加密，爆破即可。

对于每个位置 i (0-31)：

  1. 遍历所有可打印ASCII字符 (0x20-0x7E)
  2. 计算该字符在位置 i 的加权和（v为权值）
  3. 如果加权和等于 target[i]，就找到了正确的字符
  4. 输出所有32个字符组成flag

反调试绕过

可尝试动态调试验证passcode

设置调试器为 local windows debugger，Debugger->process options，设置参数为quest，开启调试。此时再打开一个cmd运行exe，会发现刚才解出的passcode报错。

此时010 editor打开的quest会提示内容被修改。

这是由于程序有反调试机制，只要一开启调试功能，quest的内容就会被修改。

main函数之前反调试就运行完了，在 Linux 中，会保存一个 init_array，它会保存一系列的函数指针，这些函数先于 main 被调用。同样的，在 windows 中也有类似的，例如本题中的 initterm 函数。

关注First指针，会发现在main之前会首先跑这两个函数1000和1190，第二个函数中的1640函数会首先进行判断，然后有一个filename参数。

利用cyberchef，还原filename小端序并异或，发现得到了quest。
